# 最小闭环开发任务拆解 v1

## 文档目的

这份文档的目标是把前面已经完成的设计文档，转化成真正可执行的开发任务清单。

到这一步，我们不再讨论“系统应该是什么”，而是明确：

- 第一阶段先做什么
- 哪些模块必须先落地
- 哪些事情可以后做
- 每个阶段的产出是什么
- 如何判断这一阶段是否完成

这份文档默认面向当前项目的最小闭环开发。

---

## 一、当前开发目标

当前阶段的目标不是直接做“最终完整产品”，而是先做出一个 **最小自动闭环**：

1. 对一段输入音频生成候选谱
2. 自动生成评测报告
3. 自动根据报告选择一个调整动作
4. 自动重跑下一轮
5. 保存多轮版本结果
6. 自动选出当前最优版本

只要这条链路跑通，我们就不再停留在“单次转谱原型”，而是进入真正的自动迭代系统阶段。

---

## 二、第一阶段开发范围

第一阶段只做下面这些最关键模块：

### 必做模块

1. 中间对象类型落地
2. 评测报告最小实现
3. 调参动作最小实现
4. 版本管理最小实现
5. 迭代引擎最小实现
6. 最小结果输出与验证

### 暂缓模块

1. 前端复杂交互增强
2. 网页内高级谱面编辑
3. 多分支并行搜索
4. 学习型调参策略
5. 高级指法优化
6. 复杂技法识别

原则很明确：

- 先把自动闭环跑通
- 再做高级体验和高级算法

---

## 三、开发任务总览

建议把第一阶段拆成 6 个开发包：

### 开发包 A：统一数据模型

目标：

- 把设计文档中的中间对象真正落成代码结构

产出：

- TypeScript 类型定义
- Python 侧输出结构定义
- 基础 JSON 文件格式统一

### 开发包 B：评测最小实现

目标：

- 先让系统能自动输出最小评测报告

产出：

- overall score
- rhythm score
- pitch score
- primaryIssues
- recommendedActions

### 开发包 C：版本管理最小实现

目标：

- 让每轮结果都有地方存，并可比较

产出：

- task 目录
- version 目录
- candidate / report / params / snapshot 文件
- bestVersionId / latestVersionId

### 开发包 D：自动调参最小实现

目标：

- 先支持几个高价值动作的自动执行

产出：

- switch_input_stem
- retune_beat_tracking
- increase_min_note_duration
- limit_pitch_range

### 开发包 E：最小迭代引擎

目标：

- 真正把一轮变多轮

产出：

- v1 -> v2 -> v3 的线性迭代
- 自动停机
- 自动选优

### 开发包 F：结果验证与回归样本

目标：

- 确保闭环不是“能跑”而已，而是“结果有改善趋势”

产出：

- 固定样本集
- 每轮结果比较
- 最小回归验证方式

## 四、开发包 A：统一数据模型

### 任务 A1：建立 TypeScript 类型

建议文件：

- `web/types/domain.ts`
- 或 `shared/types/`（如果后续要共享）

先定义这些对象：

- `InputAsset`
- `StemCandidate`
- `TimingGrid`
- `DetailedNote`
- `NotationCandidate`
- `EvaluationReport`
- `AdjustmentPlan`
- `IterationSnapshot`

### 任务 A2：建立 Python 侧对象输出格式

建议文件：

- `analyzer/types.py`
- 或 `analyzer/schemas.py`

目标：

- 让 Python 输出结构和前端消费结构一致

### 任务 A3：统一当前 analysis JSON

目标：

- 把现在的 `analysis-result.json` / `test1.analysis.json` 逐步纳入新对象体系

验收标准：

- 前后端都能用同一组核心字段理解同一份结果

---

## 五、开发包 B：评测最小实现

### 任务 B1：建立评测入口

建议文件：

- `analyzer/evaluate.py`

输入：

- 原音频
- 当前候选谱结果

输出：

- 最小评测报告 JSON

### 任务 B2：先实现最小评分项

建议第一版只实现：

- `overall`
- `metrics.rhythm`
- `metrics.pitch`
- `diagnosis.primaryIssues`
- `adjustments.recommendedActions`

### 任务 B3：生成报告文件

建议输出到：

- `output/tasks/<taskId>/versions/<versionId>/evaluation-report.json`

验收标准：

- 每跑一版候选谱，都能产出结构化评测报告

---

## 六、开发包 C：版本管理最小实现

### 任务 C1：建立任务目录结构

建议目录：

```txt
output/tasks/task_001/
  task.json
  versions/ver_001/
```

### 任务 C2：保存每轮版本文件

每版至少保存：

- `candidate.json`
- `evaluation-report.json`
- `params.json`
- `iteration-snapshot.json`
- 导出文件

### 任务 C3：维护任务级索引

在 `task.json` 中至少维护：

- `latestVersionId`
- `bestVersionId`
- `versionIds`

验收标准：

- 每轮运行后，都能在磁盘上看到完整版本记录

---

## 七、开发包 D：自动调参最小实现

### 任务 D1：实现动作字典

建议文件：

- `analyzer/adjustments.py`

第一版先支持：

- `switch_input_stem`
- `retune_beat_tracking`
- `increase_min_note_duration`
- `limit_pitch_range`

### 任务 D2：实现参数变更逻辑

目标：

- 给每个动作定义清晰的参数变化方式

### 任务 D3：实现评测 -> 动作映射

输入：

- `EvaluationReport`

输出：

- `AdjustmentPlan`

验收标准：

- 系统可以根据评测报告自动决定“下一轮改什么”

---

## 八、开发包 E：最小迭代引擎

### 任务 E1：建立引擎入口

建议文件：

- `analyzer/iterate.py`

### 任务 E2：实现线性多轮迭代

支持：

- 创建 v1
- 评测 v1
- 生成 v2
- 评测 v2
- 直到满足停机条件

### 任务 E3：实现停机条件

第一版支持：

- 达到目标分数
- 连续无提升
- 达到最大轮数

### 任务 E4：实现 best 版本更新

逻辑：

- 新版本分数更高 -> 更新 `bestVersionId`

验收标准：

- 系统能自动跑至少 2 到 3 轮
- 能自动停机
- 能自动输出 best 版本

---

## 九、开发包 F：结果验证与回归样本

### 任务 F1：建立固定样本集

建议先准备：

- 5 到 10 段代表性样本

分类建议：

- funk
- 流行
- 布鲁斯
- 难度由低到中

### 任务 F2：建立每轮结果记录表

记录：

- 版本号
- 使用 stem
- 主要动作
- 分数变化
- 是否最终入选 best

### 任务 F3：建立最小回归验证方式

最简单的第一版可以只验证：

- `overall.score` 是否提升
- `rhythm.score` 是否提升
- note 数量是否更合理

验收标准：

- 至少有一部分样本能在多轮后出现稳定提升

## 十、推荐开发顺序

如果按最稳妥路径推进，我建议顺序如下：

1. A 统一数据模型
2. B 评测最小实现
3. C 版本管理最小实现
4. D 自动调参最小实现
5. E 最小迭代引擎
6. F 固定样本回归

原因：

- 没有统一对象，后面会乱
- 没有评测，系统不会知道自己哪里错
- 没有版本管理，迭代结果没法组织
- 没有动作映射，引擎没法自动决策
- 最后才是把这些串起来跑

---

## 十一、阶段验收标准

第一阶段开发完成后，至少应满足下面这些条件：

### 验收 1

系统可以针对一个输入生成 `v1`

### 验收 2

系统可以自动输出结构化评测报告

### 验收 3

系统可以自动根据报告生成一个调整计划

### 验收 4

系统可以自动生成 `v2`

### 验收 5

系统可以自动比较 `v1` 和 `v2`

### 验收 6

系统可以在 2 到 3 轮内自动停机

### 验收 7

系统可以输出当前 best 版本

只要这 7 条满足，最小闭环就成立。
