# 人工扒谱流程分析与产品能力映射 v1

## 文档目的

这份文档用于回答一个核心问题：

- 如果一个熟练的吉他手在手工扒谱，他实际上是怎么做的？
- 这些人工步骤里，哪些可以被产品化？
- 哪些判断逻辑应该变成算法阶段，而不是简单的音频识别？

这份文档的目标不是写实现细节，而是先建立产品和算法的共同方法论。

---

## 一、我们真正要解决的问题

我们的目标不是“从音频中识别出尽可能多的音符”，而是：

- 生成一份足够接近原音频的谱子
- 让用户不需要手工修改太多
- 尤其在节奏、乐句结构、主音落点上，尽量接近人工扒谱结果

因此，这个项目本质上不是一个单纯的“音频转 note list”工具，而是一个：

- 模拟人类扒谱流程
- 自动生成候选谱
- 自动自查与迭代
- 最终输出可用谱子

的系统。

---

## 二、熟练吉他手手工扒谱时的真实流程

一个人手工扒谱通常不是按“先识别每个瞬时频率，再拼成整首谱”的方式工作。更真实的流程通常如下。

### 1. 先听整体感觉

先不急着写音符，而是先听：

- 速度大概是多少
- 是几拍子
- 重拍在哪
- groove 是直拍、shuffle、swing 还是带明显切分
- 这一段是独奏、riff、填充还是过门

这一步的本质是先建立时间框架。

### 2. 再切乐句

人不会先看每个音，而是先听句子：

- 一句从哪里开始
- 在哪里停顿
- 句尾落点在哪
- 哪些地方是呼吸点
- 哪些地方是连续跑动

也就是说，人会先把连续音流分成“有意义的乐句片段”。

### 3. 先抓骨架

在每个乐句里，人通常先抓最重要的音：

- 强拍上的音
- 长时值音
- 句尾音
- 明显落地的目标音

这些音构成了句子的骨架。即便细节没全写对，只要骨架对了，整句就开始像了。

### 4. 再补细节

在骨架基本对上之后，再去补：

- 经过音
- 装饰音
- 三连音
- 击弦、勾弦
- 滑音、推弦、揉弦

这一步不是“从零识别”，而是在上下文已知的前提下补细节。

### 5. 再做吉他化判断

就算音高一致，吉他谱也不只是“音高对”。

人还会判断：

- 这个音更可能在哪根弦
- 这句大概率在哪个把位
- 是否符合手型连续性
- 是否像某种常见 lick 或 phrasing

因此，人工扒谱的最终结果不只是音高和节奏，还包含“可演奏性判断”。

### 6. 最后回放校对

写完后，人会继续验证：

- 这份谱回放起来像不像原句
- 节奏是不是对
- 句尾是不是落稳
- 哪些音多了、少了、早了、晚了

这一步是人工扒谱里最关键的“自查闭环”。

---

## 三、人工扒谱流程的核心特点

从上面的过程可以看出，人工扒谱有几个关键特征。

### 1. 先整体，后局部

人不是先抓每个点，而是先建立：

- 速度
- 拍子
- groove
- 乐句结构

这意味着我们的系统也不应该一开始就直接输出大量 note。

### 2. 先骨架，后细节

人工扒谱时，骨架比细节重要。

如果节奏骨架、句尾落点、强拍音都错了，就算识别出很多短音，结果还是不可用。

### 3. 依赖上下文，而不是孤立识别

一个音是什么，不只由它本身决定，也由：

- 前后音关系
- 当前把位
- 当前乐句角色
- 当前节奏位置

共同决定。

### 4. 一直在校对和修正

人工扒谱是迭代行为，不是一次识别完成。

这直接说明产品必须具备：

- 自动自查
- 误差分析
- 多轮调整

## 四、把人工流程映射到产品能力

下面把人工扒谱步骤映射成产品层和算法层应有的能力。

### 阶段 A：整体节奏框架

人工行为：

- 判断 BPM、拍号、强拍、groove

产品能力：

- tempo 检测
- beat tracking
- downbeat 检测
- 小节划分
- groove 类型判定

当前状态判断：

- 我们已有 BPM 和 beat tracking
- 但还没有稳定的 downbeat 和 groove 建模
- 小节线目前仍偏机械

### 阶段 B：乐句切分

人工行为：

- 找出句子从哪开始、哪结束、哪里停顿

产品能力：

- phrase segmentation
- 停顿检测
- 音符密度变化分析
- 乐句边界候选生成

当前状态判断：

- 这一层目前还没有
- 现在系统是直接把 onset 串成 note list
- 缺少“按乐句组织”的中间层

### 阶段 C：骨架音生成

人工行为：

- 先写强拍音、长音、句尾音

产品能力：

- 主音符筛选
- 音符权重排序
- 落点音识别
- 骨架谱生成

当前状态判断：

- 这一层目前还没有
- 当前所有音符权重相同
- 这也是结果“碎”和“难用”的关键原因之一

### 阶段 D：细节补全

人工行为：

- 补经过音、装饰音、连音、技巧音

产品能力：

- ornament detection
- fast-note grouping
- triplet / syncopation 模式识别
- 技法候选层

当前状态判断：

- 当前系统实际上是直接把细节层和骨架层混在了一起
- 所以容易产生很多“识别到了，但不该直接写进谱里”的音

### 阶段 E：吉他化表达

人工行为：

- 判断把位、弦位、是否好弹

产品能力：

- 吉他指板映射
- 把位连续性评分
- 可演奏性检查
- TAB 候选生成

当前状态判断：

- 当前还几乎没有这层
- 这会导致就算 pitch 接近，结果也不像真正吉他谱

### 阶段 F：回放校对

人工行为：

- 回放听感，检查像不像原句

产品能力：

- 自动评测
- 重建音频比对
- 节奏相似度评分
- pitch contour 相似度评分
- 自动生成调试报告

当前状态判断：

- 这是后续必须补的核心系统
- 没有这一层，就无法自动优化

## 五、对当前系统的反思

从人工扒谱视角看，我们当前系统主要存在以下问题。

### 1. 过早输出细节

当前系统直接从 onset 生成 note list，导致：

- 音符过碎
- 细节音过多
- 装饰音和主音混杂

而人工扒谱不会这么做，人会先抓骨架。

### 2. 没有乐句层

当前结果缺少：

- 哪些音属于同一句
- 哪些地方是句尾
- 哪些地方是停顿

没有乐句层，就很难判断谱子“像不像一句人会写出来的话”。

### 3. 没有主次分层

当前所有 note 被平等看待。

但真实世界里：

- 有些音是骨架
- 有些音是经过音
- 有些音只是装饰或噪声

不分层，谱就会显得乱。

### 4. 缺少自查闭环

目前系统生成一版结果后，没有机制自动回答：

- 这版谱和原音频差多少
- 差在节奏、音高还是句法
- 下一轮应该调什么

所以现在的优化只能靠人工观察。

## 六、未来的目标流水线

基于人工扒谱逻辑，未来更合理的产品流水线应该是：

1. 输入视频或混合音频
2. 提取音频并分离目标吉他 stem
3. 建立 tempo、拍点、downbeat、groove 框架
4. 切分乐句
5. 先生成骨架音
6. 再补细节音和装饰信息
7. 做吉他化映射，生成 TAB + 五线谱候选
8. 将候选谱重新回放
9. 与原音频做相似度比对
10. 输出评测结果和调整报告
11. 自动迭代若干轮
12. 选出最优候选谱

这条链路的重点是：

- 从“识别音符”升级到“生成可用谱子”

---

## 七、对后续研发的指导意义

这份文档对后续研发有几个明确指导。

### 1. 优先级要重排

未来优化优先级不应是：

- 多识别几个 note

而应是：

- 先把整体节奏框架做稳
- 先把骨架做对
- 再补细节

### 2. 系统中必须加入中间层

后续不应只保留：

- beat
- measures
- notes

还应加入：

- phrases
- backbone notes
- ornament notes
- guitar fingering candidates
- evaluation report

### 3. 评测应内建到流程中

后续系统不能只“生成谱”，还必须：

- 自动验证生成结果
- 自动报告问题
- 自动支持下一轮调整

### 4. 产品定义要更明确

我们做的不是一个简单的“音频识别器”，而是一个：

- 以人工扒谱方法为核心
- 以可用谱为目标
- 以自动自查闭环为保障

的智能转谱系统。

---

## 八、当前阶段的建议结论

基于这份分析，当前最合理的研发顺序是：

1. 继续补完人工扒谱方法论
2. 补“自动评测与自查闭环设计”
3. 重构分析流程为：
   - 节奏框架层
   - 乐句层
   - 骨架层
   - 细节层
4. 再进入算法实现优化

如果这一步不先想清楚，后面很容易持续在 note 检测细节上反复调参，但结果始终离“可用谱子”很远。
